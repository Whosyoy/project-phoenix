## ADR-001：状态汇总类场景采用确定性 Workflow

- 日期：2026-07-13
- 状态：已采纳

### 背景

DirectBus 的状态汇总不是读取单一字段后直接回答。Agent 需要同时查询全局状态、构建明细和测试明细，按当前有效记录进行聚合，并正确解释全局状态与项目、基线明细之间的关系。

如果让模型自由决定查询路径或聚合逻辑，可能出现遗漏必要查询、错误使用历史记录、把单个失败状态解释为流程结束，或根据不完整证据生成状态结论等问题。

### 决策

对状态汇总类低风险场景，采用确定性 Workflow 控制查询和状态聚合，仅让模型负责意图细分、上下文抽取与自然语言解释，从而降低遗漏查询和状态幻觉风险。

确定性 Workflow 负责：

- 校验完成状态查询所需的必需参数。
- 按固定流程查询全局状态、构建明细和测试明细。
- 按已确认规则筛选当前有效记录并进行状态聚合。
- 执行数据完整性、状态一致性和异常兜底检查。
- 限制模型不得跳过必要查询或修改确定性聚合结果。

模型只负责：

- 识别状态查询下的细粒度用户意图。
- 从用户问题和已确认上下文中抽取参数。
- 根据确定性结果选择解释重点。
- 将状态、证据和 TODO 组织为清晰的自然语言回答。

### 原因

- 状态查询步骤和数据来源已经明确，适合使用固定流程保证查询完整性。
- 状态筛选与聚合属于可确定执行的业务规则，不需要模型自由规划。
- 模型擅长理解用户表达和组织解释，但不适合充当状态事实来源。
- 固定 Workflow 更容易审查查询路径、定位遗漏步骤并复现输出依据。
- 将事实计算与语言表达分离，可以减少状态幻觉和无证据推断。

### Trade-off

采用确定性 Workflow 会牺牲一部分动态灵活性：

- 新增状态字段、明细类型或查询来源时，需要显式更新 Workflow。
- 用户问题跨越状态汇总与失败诊断等多个场景时，需要进行 Intent 路由或 Skill 协作，不能在单一 Workflow 中自由扩展。
- 固定步骤可能产生额外查询成本，但换取了结果完整性和可追溯性。
- 模型可发挥的空间受到限制，但可以降低模型改变事实、遗漏查询或越权解释的风险。

### 适用范围

- 低风险、只读的状态查询与进度汇总场景。
- 查询对象、数据来源和聚合规则已经明确的场景。
- 需要同时汇总全局状态与多条项目、基线明细的场景。
- 需要基于确定性结果解释并行状态或生命周期语义的场景。
- 当前的 Pipeline Status Summary Skill。

### 不适用范围

- 需要分析失败根因、比较多种可能原因或形成诊断假设的场景。
- 需要模型动态规划未知查询路径或跨多个开放问题探索的场景。
- 重测、重构建、豁免、合入等具有副作用或需要人工确认的操作场景。
- 业务规则、数据来源或状态语义尚未确认，无法形成确定性流程的场景。
- 需要完整 RAG、Evaluation 或生产工程方案的设计问题。

## ADR-002：业务规则分层与双重校验原则

- 日期：2026-07-15
- 状态：已采纳

### 背景

在 DirectBus 场景中，规则可能出现在 Agent、Skill、Workflow、MCP Tool 和后端服务等不同层次。

尤其是重试、发起提测、发布、删除等有副作用操作，Agent 需要提前判断、解释影响并获取用户确认，Tool / 后端则必须在执行瞬间使用最新状态完成最终校验。规则归属不清会导致两个极端：

1. 所有规则都放入 Tool，使 Agent 退化为缺乏解释、降级和证据管理能力的黑盒 Tool 调用器。
2. 所有规则都放入 Agent，使权限、状态、幂等和并发等安全校验可被其他调用方绕过。

因此需要明确各层规则边界，并为有副作用操作建立不可绕过的双重校验机制。

### 决策

Project Phoenix 采用三层规则划分：

| 层次 | 核心职责 | 核心原则 |
| --- | --- | --- |
| Tool / 后端 | 保证原子能力正确、安全、一致且不可绕过 | 所有调用方都必须遵守的规则必须强校验 |
| Skill / Workflow | 保证当前业务任务的编排、证据组合、降级和结果判断正确 | 服务当前任务的编排、证据和解释 |
| Agent | 目标识别、Skill 选择、参数收集、用户交互与澄清 | 让过程可理解、可交互、可解释 |

模型只参与受限理解与表达，不作为最终安全裁判，也不能修改确定性业务事实。

### Tool / 后端规则

只要一条规则需要被所有调用方强制遵守，或绕过后会影响数据正确性、安全性和系统一致性，就必须放在 Tool / 后端。

典型规则包括：

- 参数格式和接口契约校验。
- 权限校验和鉴权。
- 幂等和重复提交校验。
- 并发控制。
- 当前状态是否允许执行。
- 操作对象是否仍然有效。
- 是否已存在执行中的同类操作。
- 最终执行前的最新状态确认。
- 接口超时、网络失败和服务不可用处理。
- 返回 Schema 校验。
- 有副作用操作的最终资格判断和实际执行。

```text
所有调用方都必须遵守
→ Tool / 后端强校验
```

### Skill / Workflow 规则

只服务当前业务任务，或依赖多个 Tool 结果进行组合判断、降级、冲突检测和结果解释的规则，应放在 Skill / Workflow。

典型规则包括：

- 当前 Skill 必须调用哪些 Tool。
- Tool 的调用顺序、依赖关系和并行条件。
- 某个 Tool 失败后是否继续或提前终止。
- 当前记录和历史记录如何过滤。
- 多个 `ToolResult` 如何转换成 `EvidenceBundle`。
- 关键证据和补充证据的定义。
- 证据缺失和证据冲突的判断。
- 何时返回 `SUCCESS`、`PARTIAL`、`UNCERTAIN` 或 `FAILED`。
- 是否需要用户确认。
- 如何解释 Tool 拒绝、失败或降级结果。

```text
服务当前任务的编排、证据和解释
→ Skill / Workflow
```

### Agent 规则

Agent 主要负责：

- 识别 Intent。
- 选择 Skill。
- 收集和补齐参数。
- 缺参时追问。
- 无法识别意图时澄清。
- 向用户解释操作影响。
- 获取用户确认。
- 将 Workflow 结果交给 ResponseGenerator 表达。

Agent 不得：

- 绕过 Tool / 后端强校验。
- 取代后端完成最终权限、状态和幂等判断。
- 基于语言推断修改确定性业务事实。
- 在证据不足时强行归因。

### 双重校验原则

对于重试、提测、发布、删除等有副作用操作，采用双层校验。

第一层是 Agent / Workflow 预检查，负责：

- 提前判断当前是否大概率允许执行。
- 减少无效 Tool 调用。
- 组合多个数据源。
- 解释为什么允许或不允许。
- 展示操作影响并获取用户确认。
- 管理证据完整性和不确定性。

第二层是 Tool / 后端最终强校验，负责：

- 使用执行瞬间的最新状态。
- 再次校验权限、幂等和业务状态。
- 防止确认期间发生的并发变化。
- 决定是否真正执行。
- 保证任何调用方都不能绕过。

例如：

```text
10:00:00 Agent 判断当前可以重试
10:00:03 其他用户已经发起重试
10:00:05 当前用户确认执行
```

最终是否允许执行，必须以后端执行时的状态为准。Agent 预检查结果不是执行授权。

该模式与前后端双重校验相同：前端或 Agent 侧检查用于提前提示、改善交互并减少无效提交；后端重新校验参数、权限、业务状态和幂等，保证规则不可绕过。

### 置信度、证据和不确定性

Agent 层优先显式表达证据，不以单一抽象的 confidence 数字代替事实状态。每个结果应尽可能说明：

- 哪些事实已经确认。
- 哪些证据缺失或冲突。
- 当前可以和不能得出哪些结论。
- 为什么返回 `PARTIAL` 或 `UNCERTAIN`。

示例：

```text
已确认：当前构建失败，失败项目为 project-a
未确认：失败根因、是否允许立即重试
原因：失败分析服务不可用
结果：PARTIAL
```

显式证据比 `confidence=0.63` 更容易解释、测试、审计和回归。

### ToolResult 与 SkillResult

`ToolResult.status` 表示一次 Tool 调用的技术状态，例如 `SUCCESS`、`TIMEOUT`、`ERROR` 或 `UNAUTHORIZED`。

`SkillResult.status` 表示整个业务任务结论的完整性和可靠性：

| SkillResult | 含义 |
| --- | --- |
| `SUCCESS` | 证据完整且一致 |
| `PARTIAL` | 部分非关键证据缺失，但已有证据仍支持部分可靠结论 |
| `UNCERTAIN` | 关键证据缺失或多个关键证据冲突，无法可靠判断 |
| `FAILED` | 整个业务能力无法执行，或关键前置条件失败 |

因此，Tool 调用失败不等于 Workflow 必须立即停止；Tool 调用成功也不等于业务结论一定可靠。

### 不同 Skill 的证据规则

不同 Skill 应具有自己的证据模型、关键证据、业务不变量、冲突规则、降级策略、前置条件、阻断规则和确认规则，不设计一套统一业务规则适配全部 Skill。

| Skill | 主要证据与规则关注点 |
| --- | --- |
| Pipeline Status Summary | 全局、构建和测试状态；当前与历史记录；阶段一致性 |
| Build Failure Analysis | 当前失败对象；`errorSummary`；`aiAnalysisResult`；失败对象关联；当前与历史失败隔离 |
| Retry Operation | 当前状态；执行中重试；对象有效性；用户确认；权限、幂等和执行后状态 |

公共执行层只处理 Tool 调用、超时、异常、Trace 和审计等技术治理。具体证据完整性、冲突判断和业务结果状态由各 Skill 的 Workflow 或 Skill 专属 Evidence Policy 负责。

### 当前内部 Retry Skill 的定位

公司内部 Retry Skill 当前更接近以下模式：

```text
Cursor
→ Retry Skill
→ 高阶 MCP Tool
→ DirectBus 后端 Workflow
```

Skill 负责意图识别、参数收集、Tool 调用和结果解释；MCP Tool / 后端负责重试资格、权限、幂等、最新状态确认和最终执行。

这不代表没有 Workflow，而是 Workflow 和大量业务规则下沉到了 MCP Tool / 后端。该模式可以称为 `Tool-backed Skill` 或“单 Tool Workflow”。

### 演进原则

当前不引入通用规则引擎，按以下顺序演进：

1. 第一阶段：规则直接写在具体 Workflow 中。
2. 第二阶段：当某个 Skill 的规则超过 3～5 条，评估抽成 Skill 专属 Evidence Policy 或 Validator，例如 `PipelineStatusEvidencePolicy`、`BuildFailureEvidencePolicy`、`RetryEvidencePolicy`。
3. 第三阶段：只有在多个 Skill 出现大量重复模式后，再评估公共 Evidence Rule 接口。

暂不引入：

- Drools。
- 通用规则 DSL。
- 万能规则平台。
- 统一业务规则引擎。

### Trade-off 与后果

- 规则分层会产生一定重复校验，但可同时获得更好的交互体验与不可绕过的安全保证。
- Skill / Workflow 可以解释和提前拒绝明显无效请求，但必须接受 Tool / 后端因最新状态变化而最终拒绝。
- 高阶 Tool 封装能复用后端能力并缩小 Agent 权限，但可能降低 Agent 侧流程可观测性，需要明确错误与证据返回契约。
- Skill 专属 Evidence Policy 会增加具体类数量，但比过早建设通用规则引擎更容易理解、测试和演进。
- 本决策定义规则归属原则，不代表当前 MVP 立即实现 Retry Skill、MCP 接入或规则引擎。

### 核心结论

1. Tool / 后端保证能力本身正确、安全且不可绕过。
2. Skill / Workflow 保证当前业务任务的编排、证据组合、降级与解释正确。
3. Agent 层负责让过程可理解、可交互、可解释；Tool / 后端负责让最终执行正确、安全且一致。
4. 对有副作用操作，Agent 预检查用于交互和解释，Tool / 后端最终校验用于保证安全和不可绕过。
