# Project Phoenix 学习与迭代治理机制

## 目的

本文用于统一 Project Phoenix 的路线控制、仓库执行、业务判断、阶段 Review 和 Sprint 验收方式，避免把仓库产出误认为学习完成，也避免脱离 DirectBus 真实业务机械推进计划。

## 角色与职责

| 角色 | 核心职责 | 决策边界 |
| --- | --- | --- |
| ChatGPT | 路线控制、阶段验收、架构 Review、学习纠偏和面试视角 | 给出路线与 Review 建议，不代替 Rory 做最终业务和学习决策 |
| Codex | 仓库操作、代码实现、源码定向分析和文档落盘 | 按已确认范围执行，不编造业务规则，不擅自扩大阶段范围 |
| Rory | DirectBus 真实业务判断、代码理解、实践验证和最终决策 | 对业务真实性、实践结果和是否采纳方案拥有最终决定权 |

三方协作原则：

- ChatGPT 控制方向和验收标准。
- Codex 将已确认的任务落实为可检查的仓库产出。
- Rory 用真实业务、代码理解和实验结果校验方案，并作最终决策。
- 存在业务事实冲突时，以 Rory 的 DirectBus 真实业务判断为准。
- 存在设计争议时，先明确假设，再通过代码、实验或证据验证。

## 阶段 Review

每次阶段 Review 必须输出以下内容：

| Review 项 | 必须回答的问题 |
| --- | --- |
| 当前阶段 | 当前处于 Phase 或哪个 Sprint？本阶段目标是什么？ |
| 完成度 | 已完成哪些验收项，哪些仍未完成？ |
| 主要短板 | 当前最影响学习质量或下一阶段推进的缺口是什么？ |
| 是否允许进入下一步 | 当前证据是否足以结束本阶段并进入下一步？ |
| 下一步唯一优先事项 | 当前只应优先完成哪一件事？ |
| 暂时不要做的事项 | 哪些内容应明确推迟，避免范围扩张？ |
| 需要固化的决策 | 哪些已验证结论需要写入 ADR、治理规则或项目文档？ |

Review 结论必须明确回答“允许进入下一步”或“不允许进入下一步”，不能只罗列产出而不作验收判断。

## Sprint 四层验收

每个 Sprint 必须同时通过以下四层验收：

### 1. 理论理解

- 能用自己的语言解释核心概念、设计选择和关键 Trade-off。
- 能说明为什么采用当前方案，以及为什么不采用主要替代方案。
- 对尚未确认的内容明确标记 TODO 或 Design Hypothesis。

### 2. 代码或实验实践

- 至少有可观察、可复现的代码、实验或验证过程。
- 能说明输入、执行过程、输出和失败现象。
- 结论必须来自实际结果，不能只依赖文档或模型生成内容。

### 3. DirectBus 业务映射

- 能说明所学能力对应 DirectBus 的哪个真实场景、用户问题或工程约束。
- 能区分已确认业务事实、待确认规则和设计假设。
- 不为了套用技术概念而编造 DirectBus 业务需求。

### 4. 简历与面试资产

- 能沉淀为可讲述的项目经历、设计决策、Trade-off 或面试问答。
- 能说明个人在分析、设计、实现和验证中的真实贡献。
- 不把尚未实现或验证的设计包装为生产成果。

只有四层验收都具有足够证据，Sprint 才能标记为完成。任一层缺失时，应在阶段 Review 中说明短板和补齐动作。

## Design Hypothesis 管理

- 未经过代码或实验验证的方案统一标记为 `Design Hypothesis`。
- `Design Hypothesis` 可以进入设计讨论，但不能作为已验证结论或生产能力对外表述。
- 实验后应根据证据将其更新为“已验证”“被否定”或“仍待验证”。
- 被实验否定的方案需要保留原因和替代方向，避免重复走弯路。

## 治理原则

- 仓库提交不等于学习完成。
- 文档写完不等于设计正确。
- MVP 跑通不等于生产可用。
- 未经过实验验证的方案标记为 `Design Hypothesis`。
- 根据实验和面试反馈动态调整路线，不机械执行初始计划。
- 每个阶段只保留一个下一步唯一优先事项。
- 暂不开展的内容必须显式记录，避免在当前 Sprint 中无边界扩张。
- 重要且可复用的架构决策应固化到 `00_Project/DECISIONS.md`。

## 执行闭环

```text
确定阶段目标
  -> 明确唯一优先事项
  -> 学习与设计
  -> 代码或实验验证
  -> DirectBus 业务映射
  -> 简历与面试资产沉淀
  -> 阶段 Review
  -> 固化决策或纠偏路线
  -> 决定是否进入下一步
```

该闭环用于约束学习和迭代节奏，不代替各 Sprint 自身的 README、TASKS、REVIEW 和 NEXT 文档。
