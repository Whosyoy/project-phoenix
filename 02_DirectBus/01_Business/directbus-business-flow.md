# DirectBus Business Flow

## Purpose

本文用于梳理 DirectBus / CI-CD 的真实业务流程，为后续 Agent 场景拆解、Tool 设计、Knowledge 设计和 Evaluation 设计提供业务输入。

当前阶段只做业务理解，不做代码实现。

## Business Positioning

DirectBus 是一个面向 Gerrit 提交的流水线工作流编排中心。

它不是简单触发单个 CI 任务，而是围绕一次提测，编排提测前准入、影响面分析、验证项目确认、冲突检测、构建验证、O 测验证和门禁合入。

DirectBus 流程可以分为两个阶段：

- 提测前: 用户数据收集、Gerrit 提交打分检查、影响面分析、最小集配置查询、验证项目确认。
- 提测后: 冲突检测、构建验证、测试验证、门禁合入。

## Pre-Test Flow

```text
用户发起提测
  -> DirectBus 收集用户数据
  -> 检查 Gerrit 提交打分
      -> 责任田主是否审批
      -> 上下游分支是否已合入成功
      -> commit message 是否包含需求 ID
      -> 其他规则 TODO
  -> 基于代码变更做影响面分析
  -> 查询最小集配置
      -> 需要验证的基线
      -> 需要验证的项目
      -> 构建参数
      -> 测试成功底版本
  -> 确认本次需要验证的项目信息
  -> 判断是否可以提测
      -> 满足准入: 创建直通车流水线
      -> 不满足准入: 拒绝提测并返回明确原因
      -> 特定场景可进入豁免审批 TODO
```

## Post-Test Flow

```text
提测成功
  -> 冲突检测
      -> 调用冲突检测 job
      -> 等待异步回调
      -> 冲突成功: 进入构建验证
      -> 冲突失败: 进入失败处理
  -> 构建验证
      -> 使用最小集配置中的测试成功底版本进行构建
      -> 按需要验证的基线项目分别触发构建
      -> 构建明细和测试明细分开存储
      -> 任意项目构建成功后，该项目可进入 O 测验证
      -> 任意项目构建失败时，全局 buildNodeStatus 可立即变为构建失败
      -> 其他项目仍继续构建 / 测试
  -> O 测验证
      -> 覆盖相机、网络、安卓、无线、底软、通信六大领域
      -> 按需要验证的基线项目分别调用 O 测异步接口
      -> 任意项目测试失败时，全局 buildNodeStatus 可立即变为测试失败
      -> 其他项目仍继续测试
      -> 测试失败:
          -> 如果错误码配置为可自动重试，最多自动重试 3 次
          -> 否则进入失败处理
      -> 正常路径下，所有需要验证的基线项目构建成功且测试成功后，进入门禁合入
      -> 豁免路径下，关键角色审批后可直接进入门禁合入，不再要求构建 / 测试全部成功
  -> 门禁合入前校验
      -> 检查分支锁是否存在
      -> 检查 patchset 是否有更改
      -> 检查 Gerrit 分数 / 审批状态是否仍满足合入
      -> 其他门禁规则 TODO
  -> 调用合入 job
      -> 等待异步回调
      -> 合入成功
      -> 合入失败: 保留已验证结果，允许人工重试合入
```

## Workflow State Fields

DirectBus 是流水线工作流，状态不能简单理解为成功 / 失败二元状态。

| Field | Meaning | Notes |
| --- | --- | --- |
| `isRunning` | 当前直通车实例是否仍处于运行生命周期 | 不是 `buildNodeStatus` 是否成功的反向值 |
| `buildStageStatus` | 当前直通车阶段 | `1`: 冲突检测，`2`: 构建，`3`: 测试，`4`: 合入 |
| `buildNodeStatus` | 当前流水线节点状态 | 全局当前状态字段，不是每个基线项目一份独立状态 |

提测前没有直通车状态。只有提测成功后，才进入 DirectBus 直通车状态体系。

### isRunning Semantics

`isRunning = true` 表示当前直通车仍可继续被处理，可能正在执行、等待回调、等待人工处理、等待重试、等待豁免或等待手动结束。

以下状态即使失败，也可能仍然保持 `isRunning = true`：

- 构建失败
- 测试失败
- 已终止
- 其他可处理阻塞状态 TODO

`isRunning = false` 目前明确出现在：

- 合入成功
- patchset 修改触发当前直通车终止
- 用户手动结束
- 其他特定规则 TODO

### buildStageStatus

| buildStageStatus | Stage | Meaning |
| --- | --- | --- |
| `1` | 冲突检测 | 当前处于冲突检测阶段 |
| `2` | 构建 | 当前处于构建阶段 |
| `3` | 测试 | 当前处于 O 测验证阶段 |
| `4` | 合入 | 当前处于门禁合入或合入 job 阶段 |

失败后 `buildStageStatus` 会停留在失败发生的阶段：

- 冲突失败: `buildStageStatus = 1`
- 构建失败: `buildStageStatus = 2`
- 测试失败: `buildStageStatus = 3`
- 合入失败: `buildStageStatus = 4`

### buildNodeStatus

| buildNodeStatus | Meaning | isRunning Relationship | Notes |
| --- | --- | --- | --- |
| 提测成功 | 直通车实例已创建，进入提测后流程 | 通常为 true | 提测前没有直通车状态 |
| 冲突检测中 | 冲突检测 job 已触发，等待回调 | 通常为 true | 真实业务状态 |
| 冲突成功 | 冲突检测通过 | 通常为 true | 进入构建验证 |
| 冲突失败 | 冲突检测未通过 | 通常为 true | 结果来自 job 回调 |
| 构建中 | 构建 job 已触发，等待回调 | 通常为 true | 使用测试成功底版本构建 |
| 构建成功 | 构建通过并输出产物 | 通常为 true | 可能表示已有项目构建成功，不等于所有项目构建完成 |
| 构建失败 | 已发现至少一个项目构建失败 | 可能为 true | 其他项目仍可能继续构建 / 测试 |
| 测试中 | O 测已触发，等待异步结果 | 通常为 true | 覆盖六大领域 |
| 测试成功 | O 测通过 | 通常为 true | 只有全部项目构建和测试成功后，正常路径才进入合入 |
| 测试失败 | 已发现至少一个项目测试失败 | 可能为 true | 其他项目仍可能继续测试 |
| 合入中 | 合入 job 已触发，等待回调 | 通常为 true | 正常路径或豁免路径都可能进入 |
| 合入成功 | 合入完成 | false | 流程完成 |
| 合入失败 | 合入 job 返回失败 | 可能为 true | 保留已验证结果，允许人工重试合入 |
| 已终止 | 当前流程被终止 | 可能为 true 或 false | patchset 修改和手动结束会导致 false |

### Global Status vs Detail Status

构建和测试不是单一节点串行执行，而是按需要验证的基线项目分别执行。

| Status Layer | Responsibility | Notes |
| --- | --- | --- |
| 全局直通车状态 | 表示当前直通车整体处于哪个阶段和节点状态 | `isRunning`、`buildStageStatus`、`buildNodeStatus` |
| 构建明细状态 | 按基线项目记录构建状态 | 与测试明细分开存储；通过 `isHistory` 区分当前记录和历史记录 |
| 测试明细状态 | 按基线项目记录测试状态 | 与构建明细分开存储；通过 `isHistory` 区分当前记录和历史记录 |

关键语义：

- 冲突检测是整个直通车的一组结果。
- 构建和测试按基线项目拆分执行。
- 任意一个项目构建失败时，全局 `buildNodeStatus` 可以立即变为构建失败，但其他项目仍会继续构建 / 测试。
- 任意一个项目测试失败时，全局 `buildNodeStatus` 可以立即变为测试失败，但其他项目仍会继续测试。
- 当某个项目构建成功后，该项目可以进入测试；此时全局 `buildStageStatus` 可能已经变为 `3`，但构建明细里仍可能有项目处于构建中。
- 全局 `buildStageStatus = 3` 或 `buildNodeStatus = 测试中` 不代表所有项目构建完成。
- Agent 回答“现在到哪一步”时，不能只看全局字段，必须结合构建明细和测试明细。

### Build-Test Association

构建和测试存在关联关系。构建成功输出产物后，DirectBus 会基于构建产物触发 O 测。

已确认关系：

- 构建明细中存在 `testId`。
- 测试明细中也存在 `testId`。
- 一个基线项目的构建结果可能关联多个测试任务。
- 多个测试任务通过多条记录表达。
- 六大领域分别有独立测试 ID: 相机、网络、安卓、无线、底软、通信。

Agent 使用规则：

- 回答“这个构建产物测了哪些领域”时，需要通过构建明细的 `testId` 和测试明细的 `domain` 关联。
- 回答“某个领域测试失败对应哪个构建产物”时，需要从测试明细反查构建明细。
- 查询当前关联关系时，应优先使用 `isHistory = 0` 的构建明细和测试明细。

### Detail Versioning with isHistory

构建明细和测试明细都通过 `isHistory` 区分当前有效记录和历史记录。

```text
重新构建 / 重新测试
  -> 将旧明细 isHistory 设置为 1
  -> copy 一条新的明细记录
  -> 新明细 isHistory 设置为 0
```

Agent 使用规则：

- 查询当前构建 / 测试状态时，应优先使用 `isHistory = 0` 的明细。
- 分析重构建 / 重测轨迹时，可以同时查看 `isHistory = 1` 的历史明细。
- 汇总整体是否可合入时，不能把历史失败记录误判为当前失败。

### Merge Entry Conditions

正常合入路径：

```text
所有需要验证的基线项目构建成功
  + 所有需要验证的基线项目测试成功
  -> 进入合入阶段
```

豁免合入路径：

```text
构建失败 / 测试失败 / 其他门禁不满足
  -> 关键角色审批豁免
  -> buildStageStatus = 4
  -> buildNodeStatus = 合入中 / 合入成功 / 合入失败
```

豁免合入不再要求所有构建 / 测试都成功。豁免规则、审批角色和审计要求单独在 `directbus-exemption-flow.md` 中细化。

| Stage | Business Question | Key Status / Output | Failure / Block Handling | Notes |
| --- | --- | --- | --- | --- |
| 用户数据收集 | 提测所需信息是否完整？ | 用户、changeId、分支、项目等 | 信息缺失则拒绝或追问 TODO | 提测前 |

| 影响面分析 | 这次变更影响哪些项目和基线？ | 影响项目、影响基线、影响领域 | 影响范围无法确认则拒绝或人工处理 TODO | 基于代码变更分析 |
| 最小集配置查询 | 需要验证哪些项目、基线和底版本？ | 验证项目、验证基线、构建参数、测试成功底版本 | 配置缺失或底版本异常 TODO | 构建输入来源 |
| 验证项目确认 | 本次提测验证范围是否明确？ | 项目、基线、领域、构建参数 | 不明确则不能提测 | TODO |
| 冲突检测 | 当前 patchset 与目标分支是否存在合入冲突？ | 冲突成功 / 冲突失败 | 调用 job，失败后进入可处理状态 | 无特殊业务推理，结果来自 job |
| 构建验证 | 影响项目是否能基于底版本构建成功？ | 按基线项目记录构建明细；全局可为构建成功 / 构建失败 | 任意项目构建失败时全局可立即标失败，但其他项目继续执行；可重构建、豁免或手动结束 | 构建侧 AI 分析结果作为证据输入 |
| O 测验证 | 构建产物是否通过六大领域测试？ | 按基线项目记录测试明细；全局可为测试成功 / 测试失败 | 任意项目测试失败时全局可立即标失败，但其他项目继续执行；错误码命中配置则最多自动重试 3 次；否则可手动重测、豁免或手动结束 | 六大领域: 相机、网络、安卓、无线、底软、通信 |
| 门禁合入前校验 | 当前验证结果是否仍可用于合入？ | 分支锁、patchset、Gerrit 分数 | patchset 变化则终止并要求重新提测；分支锁存在则暂停并允许人工重试合入 | 防止流水线期间状态漂移 |
| 合入 job | 是否最终合入成功？ | 合入成功 / 合入失败 | 合入失败保留已验证结果，允许人工重试合入 | 调用 job，异步回调同步状态 |

## Exemption / Waiver Entry Points

DirectBus 在不同业务场景下存在豁免入口。豁免不是主流程成功路径，而是失败、阻塞或规则不满足时的例外通道。

当前已知豁免可能出现在：

- 提测前准入检查失败。
- 构建失败 / 测试失败后继续推进。
- 合入前门禁不满足。

豁免需要关键角色审批。审批通过后，可以继续提测或继续合入。

具体豁免类型、审批角色、审计要求和风险等级不在本文展开，详见：

- `02_DirectBus/01_Business/directbus-exemption-flow.md`

## Evidence Available to Agent

| Stage | Evidence | Agent Boundary |
| --- | --- | --- |
| Gerrit 提交打分 | 审批状态、上下游分支状态、commit message 检查结果 | 只能解释规则和阻塞原因，不能绕过准入 |
| 影响面分析 | 影响项目、影响基线、影响领域 | 可以解释为什么触发某些验证，不能凭空改验证范围 |
| 最小集配置 | 验证项目、基线、构建参数、测试成功底版本 | 可以解释构建输入来源，不能自行选择底版本 |
| 冲突检测 | job 返回的冲突结果 | 不应猜测冲突原因，必须基于 job 结果解释 |
| 构建验证 | 失败阶段、失败模块、关键错误摘要、构建侧 AI 分析结果 | 构建侧 AI 分析结果是证据输入，不是 DirectBus Agent 的最终结论 |
| O 测验证 | 失败领域、失败用例、失败原因摘要、错误码 | 自动重试依据平台配置，Agent 不能自行判断一定可重试 |
| 门禁合入 | 分支锁、patchset 变化、Gerrit 分数 | 必须解释状态漂移和重试 / 重新提测边界 |

## Business Risks

- Gerrit 提交打分不满足时被错误放行，绕过提测准入。
- 影响面分析或最小集配置错误，导致验证项目缺失。
- 测试成功底版本同步异常或过期，导致构建输入不可靠。
- patchset 变化后继续复用旧验证结果，导致合入风险。
- O 测错误码自动重试配置误用，导致无意义重试或掩盖真实问题。
- 异步回调延迟、丢失或乱序，导致 DirectBus 状态与外部 job 状态不一致。
- 构建侧 AI 分析结果被当成最终结论，而不是证据输入。
- 豁免审批边界不清晰，导致高风险操作缺少责任人和审计链路。
