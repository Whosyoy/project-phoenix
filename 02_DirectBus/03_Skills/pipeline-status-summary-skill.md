# 流水线状态汇总技能（Pipeline Status Summary Skill）

本文定义 Sprint 1 第一个核心 Skill。采用“确定性 Workflow + 有限模型参与”，只做设计，不写代码，不定义 MCP schema，不展开完整 RAG 或 Evaluation 实现；未知规则继续标记 TODO。

## 1. 业务目标

基于指定 `apply_bus_id`，汇总 DirectBus 全局状态、当前构建明细和当前 O 测明细，回答：

- 当前直通车处于哪个阶段、节点，是否仍可继续处理。
- 哪些项目、基线已完成、执行中、待执行或失败。
- 为什么全局状态与项目明细可能同时处于不同阶段。

本 Skill 是低风险、只读能力，不负责失败根因、合入条件、重测、重构建或豁免判断。

## 2. Intent 与边界

顶层 Intent：流水线进度查询（`Pipeline Progress Query`）。

| 细粒度 Intent | 用户问题示例 | 样本 |
| --- | --- | --- |
| `QueryPipelineProgress` | “我的直通车现在到哪一步了？” | UQ-005 |
| `ExplainGlobalDetailStatus` | “为什么显示测试中，还有项目在构建？” | UQ-006 |
| `QueryPendingValidationItems` | “这个 `apply_bus_id` 还有哪些基线没跑完？” | UQ-007 |
| `ExplainPipelineLifecycleStatus` | “构建失败了，为什么 `isRunning` 还是 true？” | UQ-008 |

超出边界时应路由到其他 Skill：

- 构建或 O 测失败根因分析。
- 合入就绪或门禁诊断。
- 重测、重构建、豁免、合入等判断或操作。

## 3. 输入及缺参处理

| 输入 | 必需性 | 处理规则 |
| --- | --- | --- |
| `apply_bus_id` | 必需 | 唯一定位流水线；不得猜测或生成 |
| 用户原始问题 | 必需 | 识别细粒度 Intent 和回答重点 |
| `projectName` | 可选 | 仅在用户明确指定且能精确匹配时缩小展示范围 |
| `baselineName` | 可选 | 仅在用户明确指定且能精确匹配时缩小展示范围 |
| 已确认会话上下文 | 可选 | 仅复用唯一、明确的 ID、项目或基线 |

缺参处理：

- 缺少唯一明确的 `apply_bus_id`：停止 Tool 调用并追问。
- 存在多个可能 ID：要求用户明确选择，候选展示方式 TODO。
- 项目或基线无法精确匹配：不做模糊猜测，要求用户确认。
- “我的直通车”没有可用上下文：不假设能够自动查询最近记录，相关能力 TODO。

## 4. Workflow

查询与聚合采用固定流程，模型不得改变查询路径：

| 步骤 | 确定性动作 |
| ---: | --- |
| 1 | 校验 Intent、`apply_bus_id` 和可选展示范围；缺参则追问 |
| 2 | 调用 `query_directbus_status` 获取 `isRunning`、`buildStageStatus`、`buildNodeStatus` |
| 3 | 调用 `query_build_detail` 获取构建明细，只保留 `isHistory = 0` |
| 4 | 调用 `query_test_detail` 获取 O 测明细，只保留 `isHistory = 0` |
| 5 | 按 `projectName`、`baselineName` 分别汇总构建和测试状态 |
| 6 | 对照全局状态与当前明细，检查并行状态、数据完整性和状态冲突 |
| 7 | 根据细粒度 Intent 选择固定输出重点 |
| 8 | 基于确定性结果生成解释，并执行风险边界检查 |

固定状态规则：

- `buildStageStatus = 1 / 2 / 3 / 4` 分别表示冲突检测、构建、测试、合入。
- `isRunning = true` 表示流程仍在可处理生命周期，不代表全部任务成功。
- 构建或测试失败时，`isRunning` 仍可能为 true。
- 全局进入测试阶段时，其他项目仍可能处于构建中。
- 任一项目失败后，全局可立即显示失败，其他项目仍可能继续执行。
- 失败状态不能直接等同于流水线结束。

## 5. 模型参与点

模型可以：

- 识别四个细粒度 Intent。
- 从用户问题和已确认上下文中抽取参数。
- 选择整体进度、未完成项或状态关系等解释重点。
- 将确定性结果组织为简明自然语言。
- 在明细较多时按项目、基线、状态进行摘要。
- 识别证据不足并生成追问或转人工说明。

模型不可以：

- 跳过全局、构建或测试查询。
- 改变 `isHistory = 0` 的当前记录规则。
- 修改状态映射或确定性聚合结果。
- 猜测失败原因、待处理动作、重试资格、豁免或合入结论。
- 把空结果、Tool 失败或未知状态解释为成功。

## 6. Tool / 规则 / Knowledge

### Tool

| Tool | 职责 |
| --- | --- |
| `query_directbus_status` | 查询全局生命周期、阶段和节点状态 |
| `query_build_detail` | 查询项目、基线级构建明细 |
| `query_test_detail` | 查询项目、基线级 O 测明细 |

### 规则与 Knowledge

- `isRunning` 生命周期语义。
- `buildStageStatus`、`buildNodeStatus` 的已确认语义。
- 全局状态与构建 / 测试明细关系。
- 构建与 O 测并行流转规则。
- `buildStatus`、`testStatus` 的已确认状态语义。
- `isHistory = 0 / 1` 的当前与历史记录规则。
- 项目、基线和 `testId` 的已确认关联关系。
- 项目、基线命名规则及其他生命周期结束规则 TODO。

当前只列候选与知识类型，不定义 Tool schema 或 RAG 方案。

## 7. 输出契约

输出使用固定信息块，不定义 JSON schema：

| 输出块 | 内容 |
| --- | --- |
| 查询范围 | `apply_bus_id` 及用户明确指定的项目、基线 |
| 总体状态 | `isRunning`、当前阶段、当前节点及一句话摘要 |
| 构建进度 | 按项目、基线汇总当前构建状态 |
| 测试进度 | 按项目、基线汇总当前 O 测状态 |
| 未完成项 | 待执行、执行中、失败、取消或终止的当前记录 |
| 状态解释 | 按需解释并行状态或失败与生命周期关系 |
| 数据完整性 | Tool 是否成功、字段是否缺失、状态是否冲突 |
| TODO / 人工确认 | 无法根据现有证据确认的内容 |

先给总体结论，再给明细证据；必须区分“没有当前记录”“Tool 未返回数据”和“已经成功”。部分查询失败时只能输出部分结果。

## 8. 异常兜底

| 异常 | 兜底 |
| --- | --- |
| 全局状态查询失败 | 停止总体结论，说明查询失败；具体重试策略 TODO |
| 构建或测试明细查询失败 | 返回已取得的全局状态并标记部分结果，不判断全部完成 |
| 未查到全局状态 | 不解释为未开始、已结束或已成功，提示核对 `apply_bus_id` |
| 只有历史明细 | 不用历史记录替代当前状态，转人工核对 |
| 当前有效记录互相冲突 | 不自行选取记录，展示冲突范围并转人工 |
| 全局与明细仍无法解释 | 展示双方原始状态，标记可能不同步；处理规则 TODO |
| 未知状态值 | 保留原值并标记 TODO，不映射为已知状态 |
| 必需字段缺失 | 标记受影响结论，输出部分结果或停止对应汇总 |

## 9. 风险

风险等级：低风险，只读查询与解释。

主要风险：

- 只看全局字段，误判所有项目已经完成。
- 把失败等同于流水线结束，或把 `isRunning = true` 等同于正常执行。
- 用历史失败污染当前状态。
- 在数据不完整时给出完整结论。
- 越界进入失败诊断、重试、豁免或合入判断。

控制措施：固定查询 Workflow、当前记录过滤、数据完整性标记、模型权限限制和越界路由。
