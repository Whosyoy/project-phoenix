# 流水线状态汇总技能（Pipeline Status Summary Skill）

## 文档定位

本文定义 Sprint 1 第一个核心 Skill 的第一版设计。该 Skill 面向 DirectBus 流水线进度查询，采用“固定 Workflow + 有限模型判断”的方式组织全局状态、当前构建明细和当前 O 测明细。

当前只定义 Skill 设计，不写代码，不定义 MCP schema，不展开完整 RAG 或 Evaluation 实现。未确认的内部状态、规则和工具能力继续标记为 TODO。

## 1. 业务目标

基于指定 `apply_bus_id` 的实时数据，回答“当前直通车到哪一步了”，并在项目、基线粒度说明哪些验证项已经完成、仍在执行、尚未开始或已经失败。

该 Skill 需要解决三个核心问题：

- 正确解释 `isRunning`、`buildStageStatus` 和 `buildNodeStatus` 表达的全局状态。
- 结合 `isHistory = 0` 的构建与 O 测明细解释项目、基线级实际进度。
- 解释全局状态与明细状态并行流转时看似矛盾、实际允许出现的状态组合。

该 Skill 是只读、低风险能力，不负责构建失败根因分析、合入条件判断、重测、重构建、豁免或合入操作。

## 2. 触发 Intent

顶层 Intent：

- 流水线进度查询（`Pipeline Progress Query`）

覆盖的细粒度 Intent：

| 细粒度 Intent | 业务含义 | 对应样本 |
| --- | --- | --- |
| `QueryPipelineProgress` | 查询流水线当前阶段、节点和生命周期状态 | UQ-005 |
| `ExplainGlobalDetailStatus` | 解释全局状态与项目明细为何同时处于不同阶段 | UQ-006 |
| `QueryPendingValidationItems` | 查询尚未跑完或尚未完成的项目、基线 | UQ-007 |
| `ExplainPipelineLifecycleStatus` | 解释失败状态与 `isRunning` 生命周期的关系 | UQ-008 |

以下问题不由本 Skill 继续处理，应路由到其他 Intent / Skill：

- 构建失败的具体根因分析。
- O 测失败的具体领域、用例或错误码分析。
- 是否满足正常合入或豁免合入条件。
- 是否允许或执行重测、重构建、豁免、合入。

## 3. 典型用户问题

- “我的直通车现在到哪一步了？”
- “这个 `apply_bus_id` 现在是在构建还是测试？”
- “为什么页面已经显示测试中了，还有项目在构建？”
- “这个 `apply_bus_id` 还有哪些基线没跑完？”
- “构建失败了，为什么 `isRunning` 还是 true，这个直通车到底结束没有？”

## 4. 输入参数

| 参数 | 必需性 | 用途 | 已确认边界 |
| --- | --- | --- | --- |
| `apply_bus_id` | 必需 | 唯一定位一次 DirectBus 流水线 | 不允许模型猜测或生成 |
| 用户原始问题 | 必需 | 识别用户关注的是整体进度、并行状态、未完成项还是生命周期解释 | 只用于确定回答重点，不改变事实查询流程 |
| `projectName` | 可选 | 用户明确指定项目时缩小展示范围 | 项目命名规则仍为 TODO，不做模糊映射 |
| `baselineName` | 可选 | 用户明确指定基线时缩小展示范围 | 基线命名规则仍为 TODO，不做模糊映射 |
| 已确认的会话上下文 | 可选 | 复用当前对话中已经明确给出的 `apply_bus_id`、项目或基线 | 只有上下文中存在唯一、明确值时才能复用 |

当前状态查询固定使用当前有效明细，不把“是否查询历史记录”开放为模型自由决定的参数。重构建或重测轨迹分析不属于本 Skill 的主要职责。

## 5. 缺失参数处理

| 情况 | 确定性处理 |
| --- | --- |
| 缺少 `apply_bus_id`，且会话上下文没有唯一明确值 | 停止 Tool 调用，追问用户提供 `apply_bus_id` |
| 会话上下文存在多个可能的 `apply_bus_id` | 停止 Tool 调用，列出已知候选并要求用户明确选择；候选展示方式 TODO |
| 用户提到项目或基线，但名称不完整或无法与实时明细精确匹配 | 不做模糊猜测，返回可确认的实时范围并要求用户明确项目或基线 |
| 用户只问“我的直通车”且没有可用上下文 | 不假设系统能够自动查询“我的最近一次直通车”；相关能力未确认，标记 TODO 并追问 `apply_bus_id` |
| 用户同时提出进度查询和失败根因分析 | 本 Skill 先完成进度范围确认；失败根因部分应路由到对应诊断 Skill，不在本 Skill 内扩展 |

追问只补齐完成查询所必需的信息，不借机收集与当前进度查询无关的字段。

## 6. 固定 Workflow 执行步骤

该 Skill 默认采用固定 Workflow。模型不能改变以下步骤的先后关系，也不能因为全局状态“看起来已经足够”而跳过当前明细查询。

| 步骤 | 类型 | 固定动作 | 关键约束 |
| ---: | --- | --- | --- |
| 1 | 确定性 | 校验顶层 Intent 是否为 `Pipeline Progress Query`，确认 `apply_bus_id` 和可选展示范围 | 缺少必需参数时先追问，不调用 Tool |
| 2 | 确定性 | 调用 `query_directbus_status` 查询全局状态 | 获取并保留 `isRunning`、`buildStageStatus`、`buildNodeStatus` 的原始值 |
| 3 | 确定性 | 调用 `query_build_detail` 查询构建明细 | 当前汇总只使用 `isHistory = 0` 的有效记录 |
| 4 | 确定性 | 调用 `query_test_detail` 查询 O 测明细 | 当前汇总只使用 `isHistory = 0` 的有效记录 |
| 5 | 确定性 | 按 `projectName`、`baselineName` 汇总当前构建和测试状态 | 构建明细与测试明细分别汇总，不把历史失败混入当前结果 |
| 6 | 确定性 | 对照全局状态和当前明细做状态关系检查 | 不把全局阶段当作所有项目都已完成该阶段 |
| 7 | 确定性 | 识别已完成、执行中、待执行、失败、取消 / 终止的当前明细 | 状态只能使用已确认枚举；未知值保留原值并标记 TODO |
| 8 | 确定性 | 根据细粒度 Intent 选择对应的固定输出重点 | 查询路径不变，只调整展示重点 |
| 9 | 有限模型参与 | 基于已汇总事实生成简明解释 | 模型只能解释已确认事实和规则，不能补全缺失状态 |
| 10 | 确定性 | 执行输出前边界检查 | 确认没有给出失败根因、重试资格、豁免或合入结论 |

固定状态解释规则：

- `buildStageStatus = 1 / 2 / 3 / 4` 分别表示冲突检测、构建、测试、合入阶段。
- `isRunning = true` 表示流程仍处于可处理生命周期，不等于所有节点都执行成功。
- 构建失败或测试失败时，`isRunning` 仍可能为 true。
- 任一项目构建成功后可以进入 O 测，因此全局已经进入测试阶段时，其他项目仍可能处于构建中。
- 任一项目构建或测试失败后，全局节点状态可以立即变为失败，其他项目仍可能继续执行。
- 不能仅根据失败状态判断流水线已经结束。
- 当前明细查询必须优先使用 `isHistory = 0`；本 Skill 不用历史失败替代当前状态。

## 7. 允许模型判断的位置

模型只允许在以下位置进行有限判断：

| 判断点 | 允许模型做什么 | 不允许模型做什么 |
| --- | --- | --- |
| 细粒度 Intent 识别 | 判断用户更关注整体进度、并行状态、未完成项还是生命周期 | 创建新的顶层 Intent，或把失败根因诊断留在本 Skill 内 |
| 参数抽取 | 从用户问题和已确认会话上下文中抽取 `apply_bus_id`、项目、基线 | 猜测缺失 ID，对项目或基线做无依据的模糊映射 |
| 回答重点选择 | 根据细粒度 Intent 调整总体状态、未完成项或状态关系解释的展示顺序 | 改变固定 Tool 调用顺序或跳过明细查询 |
| 自然语言解释 | 将确定性汇总结果解释为用户可理解的进度说明 | 改写原始状态、推断未返回的下一动作或失败原因 |
| 明细摘要 | 在明细较多时按项目、基线和状态分组压缩表达 | 隐藏失败项、遗漏仍在执行的项目或把空结果解释为成功 |
| 证据不足判断 | 指出当前数据不足以支持完整结论，并生成追问或转人工说明 | 使用常识、历史印象或未知规则补齐结论 |

以下内容必须保持确定性，不交给模型自由判断：

- 是否需要查询全局状态、构建明细和测试明细。
- 是否过滤 `isHistory = 1` 的历史记录。
- 阶段码与已确认阶段含义的映射。
- 当前明细的项目、基线和状态归类。
- Tool 调用是否成功、返回字段是否缺失。
- 是否触发异常兜底或停止输出完整结论。

## 8. Tool 候选

| Tool 候选 | 在本 Skill 中的职责 |
| --- | --- |
| `query_directbus_status` | 查询指定直通车的全局生命周期、阶段和节点状态 |
| `query_build_detail` | 查询指定直通车的项目、基线级构建明细 |
| `query_test_detail` | 查询指定直通车的项目、基线级 O 测明细 |

当前只记录 Tool 候选名称和职责，不定义 MCP schema、请求参数、返回结构、超时或重试策略。真实系统名和接口边界仍为 TODO。

## 9. Knowledge 需求

本 Skill 只依赖以下知识类型：

- `isRunning` 生命周期语义。
- `buildStageStatus` 阶段码及含义。
- `buildNodeStatus` 已确认状态及含义。
- 全局状态与构建 / 测试明细的关系。
- 构建与 O 测并行流转规则。
- 构建明细 `buildStatus` 的已确认状态语义。
- 测试明细 `testStatus` 的已确认状态语义。
- `isHistory = 0 / 1` 的当前记录与历史记录规则。
- 项目、基线和 `testId` 的已确认关联关系。
- 失败后的具体可处理路径 TODO。
- 项目和基线真实命名规则 TODO。
- 未确认的其他生命周期结束规则 TODO。

当前只列知识类型，不定义知识来源接入、切分、索引、召回、排序或更新方案。

## 10. 结构化输出

输出采用固定信息块，不定义 JSON 或 MCP schema。

| 输出块 | 必需性 | 内容 |
| --- | --- | --- |
| 查询范围 | 必需 | `apply_bus_id`，以及用户明确指定的项目、基线范围 |
| 总体状态 | 必需 | `isRunning`、当前阶段、当前节点状态及一句话摘要 |
| 当前构建进度 | 必需 | 按项目、基线汇总 `isHistory = 0` 的构建状态 |
| 当前测试进度 | 必需 | 按项目、基线汇总 `isHistory = 0` 的 O 测状态；无当前测试记录时明确说明 |
| 未完成项 | 按需 | 待构建、构建中、重构建中、待测试、测试中、失败或其他未完成的当前记录 |
| 状态关系解释 | 按需 | 解释全局阶段与明细并行、失败与 `isRunning` 等关系 |
| 数据完整性 | 必需 | 说明全局、构建、测试数据是否完整，是否存在 Tool 失败、字段缺失或状态冲突 |
| TODO / 人工确认 | 按需 | 列出未知状态、未知待处理动作或无法根据现有证据确认的结论 |

输出要求：

- 先给总体结论，再给项目、基线级证据。
- 明确区分“没有当前记录”“Tool 未返回数据”和“已经成功”，三者不能互相替代。
- 存在失败项时必须保留，不因用户只问整体进度而省略。
- 存在部分 Tool 失败时，必须标记为部分结果，不能输出“全部完成”等完整性结论。
- 不展示或解释历史记录，除非明确转入其他负责历史轨迹的 Skill。

## 11. 异常与兜底

| 异常场景 | 固定兜底行为 |
| --- | --- |
| `query_directbus_status` 调用失败 | 停止生成流水线总体结论，返回查询失败事实和可重试 / 转人工提示；具体重试策略 TODO |
| 构建或测试明细 Tool 调用失败 | 可以返回已取得的全局状态，但必须标记为部分结果，不能判断所有项目是否完成 |
| 全局状态不存在 | 说明未查到指定 `apply_bus_id` 的全局状态；不把它解释为未开始、已结束或已成功 |
| 只有历史明细，没有 `isHistory = 0` 的当前明细 | 说明当前有效明细缺失，不使用历史记录代替当前状态，转人工核对数据 |
| 同一范围出现多条相互冲突的当前有效记录 | 不自行选择“更可信”的一条，展示冲突范围并转人工核对 |
| 全局状态与当前明细看似不一致 | 先应用已确认的并行流转规则；若仍无法解释，则展示双方原始状态并标记数据可能不同步，具体处理规则 TODO |
| 返回未知阶段码或未知状态值 | 保留原始值并标记 TODO，不映射为已知状态 |
| 必需字段缺失 | 标记缺失字段和受影响的结论，输出部分结果或停止对应汇总 |
| 用户问题超出进度查询范围 | 保留已确认的进度信息，并将其他问题路由到对应诊断 / 操作 Intent；不在本 Skill 内临时扩展能力 |

异步回调延迟、丢失或乱序可能导致 DirectBus 与外部任务状态暂时不一致，但本 Skill 不能在没有证据时认定具体原因。

## 12. 风险边界

风险等级：低风险，只读查询与解释。

Agent 可以：

- 查询并汇总指定直通车的全局状态和当前明细。
- 按项目、基线说明已完成、仍在执行、失败或尚未开始的验证项。
- 基于已确认状态规则解释全局状态和明细状态之间的关系。
- 在证据不完整时返回部分结果、追问或转人工建议。

Agent 不可以：

- 只看 `buildStageStatus` 或 `buildNodeStatus` 就判断所有项目已经完成。
- 把 `buildNodeStatus = 失败` 直接解释为流水线生命周期已经结束。
- 把 `isRunning = true` 解释为所有任务都在正常执行。
- 使用 `isHistory = 1` 的历史失败记录替代当前状态。
- 猜测失败根因、当前等待动作、可重试条件、豁免状态或审批结果。
- 判断是否可以合入，或建议绕过构建、O 测、门禁和审批。
- 触发重测、重构建、豁免、合入、结束流水线等有副作用操作。
- 在 Tool 失败或数据不完整时给出“全部完成”“已经结束”等完整结论。

## 13. 后续 Evaluation 方向

当前只定义评测方向，不编写评测代码或完整数据集。

| 评测方向 | 重点检查 |
| --- | --- |
| 顶层 Intent 识别 | 是否稳定识别为 `Pipeline Progress Query`，并避免误路由到失败诊断或操作 Intent |
| 细粒度 Intent 识别 | 是否区分整体进度、全局与明细解释、未完成项和生命周期解释 |
| 参数抽取与追问 | 是否正确提取 `apply_bus_id`；缺失或歧义时是否停止查询并追问 |
| 固定 Workflow 遵循 | 是否按固定顺序查询全局状态、构建明细和测试明细，避免只查全局状态 |
| Tool 选择准确性 | 是否只选择本 Skill 所需的三个查询 Tool，避免调用副作用 Tool |
| 当前记录准确性 | 是否只使用 `isHistory = 0` 汇总当前状态，并避免历史失败污染结果 |
| 项目 / 基线汇总完整性 | 是否完整列出仍在构建、测试、等待或失败的项目和基线 |
| 状态语义准确性 | 是否正确解释阶段码、节点状态和 `isRunning` 生命周期 |
| 并行状态解释 | 是否识别“全局测试中但部分项目仍在构建”等合法组合 |
| 失败与结束区分 | 是否避免把构建 / 测试失败等同于流水线结束 |
| 结构化输出完整性 | 是否包含总体状态、构建进度、测试进度、数据完整性和必要 TODO |
| Tool 异常兜底 | 部分或全部查询失败时，是否降级为部分结果或停止结论 |
| 未知规则处理 | 是否保留未知状态与内部规则 TODO，而不是编造映射或待处理动作 |
| 安全边界 | 是否保持只读，不触发重试、重构建、豁免、合入或结束操作 |

后续样本应优先覆盖 UQ-005～UQ-008，并补充缺少 `apply_bus_id`、明细 Tool 部分失败、只有历史明细、未知状态值和全局 / 明细无法解释等边界场景。
