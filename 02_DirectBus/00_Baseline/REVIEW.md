# Phase 0 复盘：DirectBus 业务基线（Business Baseline）

## Phase 0 目标

Phase 0 的目标是梳理 DirectBus / CI-CD 真实业务流程，明确适合由智能体（Agent）支撑的业务问题、所需证据和安全边界，为后续 Agent Core、Skill 与 Tool 设计建立可信的业务输入。

本阶段坚持业务优先、先理解再实现，不编写业务代码，不绑定具体 Agent 框架，也不实现完整的 MCP、RAG 或 Evaluation。

## 已建立的业务基线

### 1. DirectBus 主链路

DirectBus 是面向 Gerrit 提交的流水线工作流编排中心，主链路已经明确为：

```text
提测前准入
  -> 影响面分析与最小集配置
  -> 验证项目确认
  -> 冲突检测
  -> 构建验证
  -> O 测验证
  -> 门禁校验
  -> 合入
```

### 2. 状态与明细关系

- `isRunning` 表达直通车是否仍处于可处理生命周期，不等于节点是否成功。
- `buildStageStatus` 表达流水线当前阶段，`buildNodeStatus` 表达全局节点状态。
- 构建和测试按项目、基线拆分执行，全局状态不能替代明细状态。
- 查询当前构建和测试结果时，优先使用 `isHistory = 0` 的当前有效明细。
- 分析重构建或重测轨迹时，才结合 `isHistory = 1` 的历史明细。
- 单个项目失败后，其他项目仍可能继续构建或测试。

### 3. 正常路径与例外路径

- 正常合入要求所有需要验证的基线项目构建成功且测试成功。
- 豁免是失败或规则不满足时的例外通道，不是普通成功路径。
- 豁免需要关键角色审批并保留证据；具体规则、审批角色和审计要求仍有 TODO。
- PatchSet 变化、影响范围未知、底版本不可用当前都不能默认视为可豁免。

### 4. 证据优先原则

- 状态查询必须以 DirectBus 全局状态和当前明细为依据。
- 构建诊断必须引用失败阶段、失败模块、错误摘要或报告等证据。
- 构建侧 AI 分析结果只能作为辅助证据，不能直接作为最终结论。
- O 测诊断必须引用失败领域、失败用例、错误码和报告等证据。
- 门禁诊断必须基于分支锁、PatchSet、Review 分数和已知门禁结果。
- 规则、权限或审批信息未知时必须保留 TODO，不能推测补全。

| 顶层意图 | 覆盖问题 | 核心业务目标 |
| --- | --- | --- |

## 当前已确认的 Agent 边界

Agent 可以：

- 查询和解释 DirectBus 实时状态、当前明细、规则与证据。
- 汇总项目、基线、构建、测试、门禁和豁免相关信息。
- 诊断阻塞与失败，给出基于证据的补齐或处理建议。
- 在已知规则范围内检查重测、重构建等操作的前置条件。
- 生成建议或豁免申请草稿，但草稿不代表审批结果。

Agent 不可以：

- 只根据用户描述或单个全局状态字段直接下结论。
- 绕过提测准入、门禁或关键角色审批。
- 无确认触发重测、重构建等有副作用操作。
- 自动批准豁免、强制合入或复用已经失效的验证结果。
- 把构建侧 AI 分析结果当作最终事实。
- 在规则、权限、影响范围或证据不明确时编造结论。

## 当前仍然保留的 TODO

- 用户提测参数、提测分数要求和其他 commit message 格式规则。
- 影响面分析服务的真实输入、输出和异常处理。
- 最小集配置内容、底版本选择及底版本同步失败处理。
- 冲突检测失败后的人工处理和重试规则。
- 构建失败类型、可重构建条件、次数限制、权限和可豁免规则。
- O 测错误码说明、手动重测条件、次数限制和权限规则。
- 完整门禁规则、合入失败分类和人工重试合入条件。
- 豁免入口、可豁免 / 不可豁免规则、审批角色、证据要求、有效期和审计要求。
- Tool 候选的真实系统名、能力范围和接口边界。
- Evaluation 黄金数据集规模、样本格式和通过标准。

这些 TODO 是当前业务理解的边界记录，不在 Phase 0 中猜测补全。

## Phase 0 的业务与项目价值

Phase 0 将 DirectBus Agent 从抽象想法收敛为一组可追踪的业务资产：真实流程、状态语义、异常路径、30 条用户问题、8 个顶层意图、候选 Skill / Tool / Knowledge 以及统一安全边界。

这些产出为下一阶段提供了共同语言和设计约束，使 Agent Core 与 Skill / Tool 设计可以从已确认的业务问题出发，并在遇到未知内部规则时保持 TODO，而不是提前进入框架、接口或实现细节。
